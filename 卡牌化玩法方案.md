# TEC3 卡牌化玩法方案

> 将剧情编辑工具中的玩法机制（命令系统、反应系统、对话体系、隐藏线索、场景绑定等）以卡牌交互方式实现的设计文档。

---

## 一、设计目标

在 TEC3 现有卡牌体系基础上，通过**扩展卡牌类型**和**丰富交互方式**，完整覆盖剧情编辑工具中描述的所有玩法机制，让游戏世界完全由卡牌驱动。

### 核心原则

- **一切皆卡牌**：所有剧情元素（物品、人物、场景、线索、选项、规则）都以卡牌形式存在
- **卡牌即操作**：通过拖放、堆叠、检视、点击等卡牌交互触发玩法
- **状态驱动生成**：卡牌的获取、显示、消失由 flag 和线索条件驱动

## 二、卡牌类型扩展（5 类 → 8 类）

| 类型 | CSS 类 | 来源 | 主要交互 | 视觉特征 |
|------|--------|------|----------|----------|
| 道具卡 | `item-card` | 编辑器物品 | 双击检视、堆叠、出示 | 基础卡牌 |
| 场景卡 | `scene-card` | 编辑器场景 | 双击进入场景 | 较大尺寸、场景图标 |
| 人物卡 | `char-card` | 编辑器角色 | 双击对话 | 头像占主体 |
| 线索卡 | `clue-card` | 编辑器线索 | 归因推演、解锁 | 三色边框（视觉/听觉/味觉） |
| 功能卡 | `logic-card` | 系统 | 归因、回收 | 系统图标 |
| **密码锁卡** | `lock-card` | **新增** | 拖入正确卡牌解锁 | 锁图标 + 可见凹槽 |
| **分支卡** | `choice-card` | **新增** | 点击选项 | 多选项按钮列表 |
| **事件卡** | `event-card` | **新增** | 条件触发后自动出现 | 闪光、出现后自毁 |

## 三、6 大卡牌交互引擎

### 3.1 拖放堆叠 StackEngine（已存在 → 扩展）

**职责**：物品 + 物品 = 组合/合成/使用

**扩展字段**（在 `js/config/cards.js`）：

```js
{
  id: 'ITEM_a',
  name: '物品A',
  type: 'item',
  class: 'item-card',
  
  // 已有字段
  desc: '物品描述',
  consumable: true,
  
  // 新增：useItem 命令支持
  useItem: {
    requiredItems: ['ITEM_b'],           // 需要哪些物品
    successFlags: ['FLAG_已开锁'],         // 成功后设置的标记
    successMessage: '成功解锁！',          // 成功提示
    failMessage: '这个物品没有反应',       // 失败提示
    obtainItem: 'ITEM_new',              // 成功后获得的卡牌
    jumpScene: 'SCENE_next',             // 成功后跳转的场景
    consumeOnSuccess: true,              // 是否消耗材料
  }
}
```

**匹配规则**：
- 拖放 A + B 触发 A 的 useItem（A 是"主物品"，B 是"消耗品"）
- 匹配 `requiredItems` 数组中的物品 ID
- 匹配失败 → 触发 `failMessage` 并显示文本气泡
- 匹配成功 → 设置 flag、生成新卡、跳转场景

### 3.2 双击检视 InspectEngine（新增）

**职责**：双击物品卡 → 展示文本 + 设置 flag

**触发条件**：
- 双击物品卡（`item-card`）
- 物品卡包含 `text` 命令字段

**数据结构**：

```js
{
  id: 'ITEM_diary',
  name: '日记本',
  type: 'item',
  class: 'item-card',
  desc: '一本破旧的日记',
  
  // 检视命令
  inspect: {
    text: '日记里写道："那天晚上我们看到了不该看到的东西..."',
    setFlags: ['FLAG_已读日记'],
    revealClue: 'CLUE_diary_entry',     // 检视后揭露的隐藏线索
    markAsInspected: true,              // 标记为已检视
  }
}
```

**UI 表现**：
- 未检视：卡牌右上角显示 🔍 图标
- 已检视：🔍 图标变为 ✓ 标记，卡牌边框变深
- 检视时：弹出气泡显示文本
- 特殊文本：支持 `[char:ID]`、`[item:ID]`、`[clue:ID]` 引用语法（自动高亮为彩色标签，点击跳转）

**与 flag 系统联动**：
- 检视设置 `FLAG_已读日记` 后，`conditional` 规则会重新评估
- 可能触发新卡牌生成或对话内容变化

### 3.3 拖入槽位 SlotEngine（已存在 → 扩展）

**职责**：拖入卡牌到目标卡牌的槽位 → 触发对应玩法

**当前用途**：探索窗口、对话窗口、归因推演窗口

**扩展用途**：

#### 3.3.1 密码锁卡（新增）

**卡牌定义**：

```js
{
  id: 'LOCK_safe',
  name: '保险箱',
  type: 'lock',
  class: 'lock-card',
  desc: '一个老式保险箱，需要三个数字',
  
  // 密码配置
  password: {
    slots: 3,                          // 槽位数
    hint: '密码与五行相关',             // 提示文本
    correctItems: ['ITEM_wuxing_1', 'ITEM_wuxing_2', 'ITEM_wuxing_3'],
    successMessage: '保险箱打开了！',
    successFlags: ['FLAG_safe_opened'],
    obtainItem: 'ITEM_evidence',
    failMessage: '密码错误',
  }
}
```

**UI 设计**：
- 卡牌本体上直接绘制 3 个凹槽（类似传统密室逃脱）
- 玩家将线索卡/物品卡拖入凹槽
- 凹槽被填满后自动判定
- 也可保留"打开弹窗"模式用于复杂密码

#### 3.3.2 物品出示（presentReactions）

**交互方式**：将任意卡牌拖到**物品卡**的"出示区"（非堆叠区域，而是物品卡底部的小型高亮区域）

**数据结构**：

```js
{
  id: 'ITEM_altar',
  name: '祭坛',
  type: 'item',
  class: 'item-card',
  
  // 出示反应
  presentReactions: [
    {
      triggerItem: 'ITEM_incense',
      conditionFlags: ['FLAG_已开灯'],
      message: '香炉开始冒烟...',
      setFlags: ['FLAG_offering_accepted'],
      obtainItem: 'CLUE_smoke_pattern',
    },
    {
      triggerItem: 'ITEM_fruit',
      message: '祭坛不接受这个',
      setFlags: null,
    }
  ]
}
```

**UI 表现**：
- 物品卡底部有一个虚线框"拖入这里出示"
- 拖入卡牌后立即触发对应反应（不消耗卡牌）
- 多个反应按顺序匹配，命中后停止

#### 3.3.3 对话槽扩展（角色卡）

**数据结构**：

```js
{
  id: 'CHAR_witness',
  name: '目击者',
  type: 'char',
  class: 'char-card',
  
  // 多层对话体系
  dialogue: {
    // 优先级 1：线索反应
    clueReactions: [
      {
        triggerClue: 'CLUE_photo',
        message: '这张照片...我想起来了！那天晚上...',
        setFlags: ['FLAG_witness_recalled'],
        nextDialogue: 'witness_full_testimony',
      }
    ],
    
    // 优先级 2：物品反应
    itemReactions: [
      {
        triggerItem: 'ITEM_gun',
        message: '那是凶器！别再拿出来！',
        setFlags: ['FLAG_witness_afraid'],
      }
    ],
    
    // 优先级 3：场景对话
    sceneTalks: [
      {
        sceneId: 'SCENE_crime_scene',
        talks: ['你在这里做什么？', '这不是你该来的地方。'],
      }
    ],
    
    // 优先级 4：默认对话
    defaultTalks: [
      '我什么都不知道。',
      '让我静一静。',
      '不要再问了。',
    ],
  }
}
```

**匹配优先级**：
1. 拖入的是线索卡 → 匹配 `clueReactions`
2. 拖入的是物品卡 → 匹配 `itemReactions`
3. 当前活跃场景匹配 → 显示 `sceneTalks`
4. 兜底 → 显示 `defaultTalks`

**与对话窗口集成**：
- 现有对话窗口的 1 个槽位保持不变
- 拖入卡牌后，按优先级匹配最合适的对话内容
- 对话内容中可包含"对话选项"（→ 分支卡）

### 3.4 点击选项 ChoiceEngine（新增）

**职责**：分支卡（choice-card）上点击选项 → 触发对应结果

**卡牌定义**：

```js
{
  id: 'CHOICE_confront',
  name: '对峙',
  type: 'choice',
  class: 'choice-card',
  
  // 选项配置
  choices: [
    {
      label: '指出证据',
      requiresFlags: ['FLAG_已收集证据'],
      successMessage: '嫌疑人哑口无言',
      setFlags: ['FLAG_confronted_with_evidence'],
      obtainItem: 'CLUE_confession',
      jumpScene: 'SCENE_ending_truth',
    },
    {
      label: '直接质问',
      requiresFlags: null,            // 无前置条件
      successMessage: '嫌疑人开始狡辩',
      setFlags: ['FLAG_confronted_directly'],
      jumpScene: 'SCENE_ending_ambiguous',
    },
    {
      label: '沉默离开',
      requiresFlags: null,
      successMessage: '你选择暂时离开',
      setFlags: ['FLAG_chose_silence'],
    }
  ]
}
```

**UI 设计**：
- 方案 A：**分支卡直接显示** —— 卡面上列出 3~4 个可点击按钮
- 方案 B：**对话窗口内显示** —— 对话流中出现选项，悬浮在气泡旁
- 玩家点击选项 → 选项消失 → 触发对应效果

**条件选项**：
- 选项可设置 `requiresFlags`，不满足时按钮变灰或隐藏
- 实现"看过某物后才能选择"的剧情限制

### 3.5 后台触发 TriggerEngine（新增）

**职责**：条件规则（`conditional` 命令）后台轮询 → 自动触发效果

**核心机制**：
- 不生成可见卡牌（除非需要生成新卡）
- 每次玩家操作后（检视/堆叠/对话/探索）触发评估
- 评估所有注册的条件规则，命中后执行对应效果

**数据结构**：

```js
{
  id: 'RULE_reveal_basement',
  type: 'rule',                          // 后台规则，不显示
  
  // 触发条件
  condition: {
    flags: ['FLAG_已读日记', 'FLAG_已找到钥匙'],  // 必须都已设置
    clues: ['CLUE_血字'],                          // 必须已收集
    items: ['ITEM_flashlight'],                    // 必须持有
  },
  
  // 触发效果
  effect: {
    spawnCard: {                          // 生成新卡牌到桌面
      type: 'item',
      template: 'ITEM_basement_key',
      position: { x: 300, y: 200 },
      message: '你突然想起了什么...',
    },
    setFlags: ['FLAG_basement_unlocked'],
  },
  
  // 规则属性
  oneTime: true,                          // 只能触发一次
  triggerMessage: '阁楼的门似乎可以打开了',
}
```

**评估时机**：
- 玩家进行任何操作后
- 每次评估后标记已触发的规则（避免重复）
- 满足条件时立即触发，无需玩家操作

**与现有 triggerEnding 的区别**：
- `triggerEnding`：单一结果（结局触发）
- `TriggerEngine`：通用条件引擎，可触发任意效果

### 3.6 双击进入场景 SceneEngine（已存在 → 扩展）

**职责**：场景卡 = "场景容器"，双击进入切换桌面

**核心概念**：
- 不再是单一桌面，而是"主桌面 + 多个场景容器"
- 双击场景卡 → 切换到该场景的专属面板
- 场景面板内只显示该场景的卡牌
- 提供"返回主桌面"按钮

**数据结构**：

```js
{
  id: 'SCENE_basement',
  name: '地下室',
  type: 'scene',
  class: 'scene-card',
  desc: '阴暗潮湿的地下室',
  
  // 场景卡牌组
  cards: [
    'ITEM_old_chest',
    'CHAR_ghost',
    'CLUE_blood_stain',
  ],
  
  // 场景配置
  exploration: {
    slots: 3,
    duration: 5000,
    drops: [
      { id: 'CLUE_铁锈的味道', chance: 0.5, guaranteed: true },
    ],
  }
}
```

**UI 表现**：
- 场景卡双击 → 当前桌面淡出 → 场景面板淡入
- 场景面板显示该场景内的所有卡牌
- 顶部有"返回主桌面"按钮
- 探索系统保留，但只在场景面板内生效

**优势**：
- 解决"卡牌过多导致桌面混乱"的问题
- 场景成为独立的"小关卡"
- 符合剧情推进的节奏

## 四、Flag 系统（条件基础）

### 4.1 状态扩展

`state.js` 从单一字段扩展为完整状态系统：

```js
export const gameState = {
  // 已有
  hasSanityPollution: false,
  sceneSynced: false,
  isGameOver: false,
  lastTriggeredClueId: null,
  
  // 新增
  flags: new Set(),           // 标记集合
  knownClues: new Set(),      // 已收集线索
  inspectedItems: new Set(),  // 已检视物品
  unlockedItems: new Set(),   // 已解锁物品
  currentScene: null,         // 当前活跃场景ID
  triggeredRules: new Set(),  // 已触发的规则（防止重复）
};
```

### 4.2 Flag API

```js
// 标记操作
setFlag(name)                   // 设置标记
hasFlag(name)                   // 检查标记
clearFlag(name)                 // 清除标记

// 线索操作
addClue(id)                     // 添加线索
hasClue(id)                     // 检查线索

// 检视记录
markInspected(itemId)           // 标记已检视
isInspected(itemId)             // 检查是否检视
```

### 4.3 Flag 命名规范

- 全大写下划线：`FLAG_已读日记`、`FLAG_safe_opened`
- 分类前缀：人物类 `FLAG_CHAR_...`、场景类 `FLAG_SCENE_...`、物品类 `FLAG_ITEM_...`

## 五、引用语法（可选增强）

### 5.1 文本引用

支持在描述、对话、文本气泡中使用 `[type:ID]` 引用其他实体：

```
"你看到了 [char:CHAR_witness] 站在 [loc:SCENE_basement] 入口，手里拿着 [item:ITEM_gun]。"
```

### 5.2 渲染规则

- `[char:ID]` → 绿色标签
- `[item:ID]` → 黄色标签
- `[clue:ID]` → 红色标签
- `[loc:ID]` → 蓝色标签

### 5.3 点击交互

- 点击标签 → 对应卡牌高亮闪烁
- 镜头平滑移动到该卡牌位置
- 形成"文字引导"效果

## 六、隐藏线索机制

### 6.1 卡牌级隐藏

```js
{
  id: 'CLUE_secret_door',
  name: '暗门',
  type: 'clue',
  class: 'clue-card',
  desc: '墙壁上的一个隐蔽入口',
  
  // 隐藏条件
  revealCondition: {
    flags: ['FLAG_已读日记'],
    clues: ['CLUE_blood_stain'],
  },
}
```

### 6.2 场景级隐藏

```js
{
  id: 'SCENE_basement',
  name: '地下室',
  cards: ['ITEM_old_chest', 'CHAR_ghost'],
  
  // 场景内可揭示的隐藏线索
  hiddenClues: [
    {
      id: 'CLUE_basement_inscription',
      revealChance: 0.3,
      requiresFlags: ['FLAG_flashlight_on'],
    }
  ]
}
```

### 6.3 触发流程

1. 玩家进行探索/检视/对话操作
2. 系统检查是否满足 `revealCondition`
3. 满足则自动生成新卡牌到桌面（带入场动画）
4. 一次性触发，避免重复

## 七、整体卡牌玩法架构图

```
玩家操作
  │
  ├── 双击 ──────── InspectEngine ──── 检视物品 / 文本命令
  │
  ├── 拖放堆叠 ─── StackEngine ────── useItem 命令 / 组合
  │
  ├── 拖入槽位 ─── SlotEngine ─────── 探索 / 对话 / 归因 / 出示 / 密码锁
  │
  ├── 点击选项 ─── ChoiceEngine ───── choice 命令 / 分支
  │
  ├── 场景切换 ─── SceneEngine ────── 进入 / 返回场景
  │
  └── 操作完成 ─── TriggerEngine ──── conditional 规则 / 隐藏线索
                         ↓
                    更新 gameState
                    检查所有规则
                    生成新卡牌
                    触发弹窗/气泡
```

## 十、关键设计原则

1. **卡牌即世界**：所有可交互的剧情元素都是卡牌，没有隐藏的"机关"或"按钮"
2. **状态驱动**：所有卡牌的出现、消失、可用性都由 gameState 决定
3. **6 种交互穷举**：双击、拖放、拖入、点击、场景切换、后台触发，覆盖所有玩法
4. **可视化配置**：所有玩法逻辑都通过卡牌字段配置，无硬编码
5. **可扩展性**：新增玩法只需新增卡牌类型或扩展卡牌字段

## 十一、真名揭示系统 v2（待实现）

### 11.1 当前设计（即将废弃）

当前真名卡的运作方式：
- 玩家将五感线索卡（CLUE_vision_eye 等）拖到 `???` 卡上 → 标记该感官已收集
- 5 种感官全部收集后 → 揭示真名「深渊低语者」

**问题**：玩家主动收集，缺乏紧迫感和对抗性，五感线索卡用完即闲置。

### 11.2 v2 设计：泄漏式对抗

#### 核心机制

```
真名卡每 60s 泄漏一张感官碎片 → 碎片有 1h 存活时限
       ↓
玩家需用对应五感线索卡清除碎片
       ↓
5 张碎片全部清除 → 真名揭示
       ↓
最后一张碎片超时未清除 → 游戏失败
```

#### 流程拆解

```
t=0     真名卡出现在桌面
t=60s   泄漏 #1 感官碎片（如 CLUE_shard_vision）→ 出现在桌面
t=120s  泄漏 #2 感官碎片（如 CLUE_shard_hearing）
t=180s  泄漏 #3 感官碎片
t=240s  泄漏 #4 感官碎片
t=300s  泄漏 #5 感官碎片 → 停止泄漏（共 5 张）
         ↓
每张碎片倒计时 3600s（1h），超时则游戏失败
         ↓
玩家需要拖对应五感线索卡（CLUE_vision_eye 等）到碎片上
→ 二者都消耗（consumable: true）
→ 该感官标记为已清除
         ↓
5 张碎片全部清除 → ??? 卡揭示为「深渊低语者」
```

#### 特殊规则

1. **泄漏顺序固定**：vision → hearing → taste → touch → smell（可配置）
2. **清除顺序不固定**：玩家可在任意碎片泄漏后立即清除（如先清 hearing，再等 vision 泄漏再清）
3. **超时判定**：每张碎片独立倒计时，最后一张超时 → 游戏失败（不分先后清除的碎片）
4. **清除消耗**：五感线索卡 + 感官碎片 **都销毁**（玩家需要重新获取五感线索卡）
5. **五感线索卡获取**：玩家可通过探索/捕获/组合重新获得用完的感官线索

#### 数据结构

## 十三、腐化与时间压力系统（待实现）

### 13.1 设计目标

在卡片数量增多后，需要给玩家施加适度的紧张感和目标导向。采用 **"卡牌腐化 + 蔓延"** 的混合机制。

### 13.2 核心机制

**腐化倒计时**
- 关键卡牌掉落时自动附加腐化倒计时（默认 2-3 分钟）
- 卡面出现环形倒计时条，随时间消减
- 剩余 30 秒时卡牌开始红色脉冲闪烁，压感递增
- 倒计时归零 → 卡牌腐化消失或被替换为"腐化残渣"

**腐化蔓延**
- 腐化归零的卡牌不会直接消失，而是先向邻近卡牌扩散"感染"
- 被感染的卡牌获得额外的腐化标记
- 腐化卡需要先使用净化/消耗类操作才能恢复正常交互
- 如果整片区域都被感染，形成"死区"——该区域的卡牌全部失效

**压力梯度设计**
| 阶段 | 时间 | 视觉效果 | 玩家感受 |
|------|------|----------|----------|
| 正常 | 倒计时 > 60s | 细光环 | 有目标，可控 |
| 警告 | 30-60s | 光环变粗、橙黄色 | 需要行动 |
| 危机 | < 30s | 红色脉冲、卡面微颤 | 紧迫感 |
| 腐化 | 0s | 黑烟扩散、邻近卡变色 | 后果已发生 |
| 蔓延 | 0s+ | 感染标记扩散 | 连锁压力 |

### 13.3 适合施加腐化的卡牌

- `ITEM_shadow_figure`（黑影）→ 需要在限定时间内用 `ITEM_sdt` 照亮
- `ITEM_true_name`（???）→ 揭示前会向周围泄漏
- `CLUE_tx`（铁锈）→ 持续污染，需要快速归因
- 未来新增的关键剧情卡

### 13.4 对抗手段

- 使用组合消灭腐化卡（完成叙事目标即消除压力）
- `ITEM_coin`（因果律）可延缓倒计时（消耗 1 枚 +30s）
- `LOGIC_capture` 捕获后冻结腐化——但捕获的卡牌使用后可能会重新启动倒计时

### 13.5 视觉设计

```
正常状态：
┌─────────────────┐
│░░░░░░░░░░░░░░░░░│  ← 细光环（颜色随倒计时变化）
│                 │
│    [黑影]       │
│                 │
└─────────────────┘

危机状态（<30s）：
┌─────────────────┐
│▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓│  ← 红色脉冲粗光环
│   ⚠ 腐化⚠      │  ← 卡面微颤动画

```

### 13.6 实施优先级

1. 腐化倒计时光环 + 计时器 [高]
2. 红色脉冲警告动画 [高]
3. 倒计时归零腐化逻辑 [高]
4. 因果律延缓倒计时 [中]
5. 腐化蔓延感染 [中]

## 十四、理智与负面卡系统（设计构思）

### 14.1 设计目标

引入"理智值"作为全局压力资源，与探索收益形成**自我加速的负反馈循环**——探索越多、收益越高，但理智越低、负面卡越密集，最终在 0 理智时游戏结束。让压迫感不依赖于剧情脚本，而是从系统层面自然产生。

### 14.2 理智卡

**形态**：固定在棋盘边缘（或 header 区）的独立卡牌，不参与拖拽/堆叠。

**显示**：
```
┌─────────────┐
│   🧠 理智   │
│   100 / 100  │  ← 数值变化时跳动 + 变色
│  ████████░░  │  ← 进度条
└─────────────┘
```

**数值变化**：
| 触发条件 | 变动 | 视觉反馈 |
|----------|------|----------|
| 探索完成 | -10~-30（按场景风险） | 数字跳动，进度条收缩 |
| 负面卡未被清除持续在场 | -1/s（持续） | 缓慢衰减 |
| 净化成功 | +15~+30 | 绿色闪光 |
| 归零 | 游戏结束 | 全屏效果 |

### 14.3 负面卡产生（双路线）

#### 路线 A：探索掉落

在场景 drops 中直接配置负面卡模板。风险越高的场景，负面卡生成概率越高。

```js
// 场景探索配置扩展
drops: [
  // 现有掉落
  { id: 'ITEM_something', chance: 0.6, guaranteed: false },
  // 负面卡掉落（高收益场景概率更高）
  { id: 'DEBUFF_anxiety', chance: 0.3, negative: true },
  { id: 'DEBUFF_paranoia', chance: 0.1, negative: true },
]
```

#### 路线 B：定时弹出（核心压迫机制）

**规则**：
- 每 60s/90s 随机向桌面投一张负面卡
- 弹出间隔随理智降低而缩短（100→50 时间隔减半）
- 不依赖任何玩家操作，纯定时驱动
- 制造真正的压迫感

```js
// 弹出间隔 = 基础间隔 × (当前理智 / 最大理智)
// 基础间隔 60s：100 理智时 → 60s，50 理智时 → 30s，25 理智时 → 15s
间隔(ms) = 60000 × (sanity / 100)
```

### 14.4 负面卡类型（示例）

| ID | 名称 | 效果 | 清除方式 |
|----|------|------|----------|
| `DEBUFF_anxiety` | 焦虑 | 占用一个棋盘位，−1/s 理智 | 休息 / 对话净化 |
| `DEBUFF_paranoia` | 偏执 | 禁止相邻卡牌交互 | 链式消除 → 恐惧 |
| `DEBUFF_fear` | 恐惧 | 每 10s 随机移动自身位置 | 链式消除 → 愤怒 |
| `DEBUFF_anger` | 愤怒 | 禁止探索操作 | 链式消除 → 平静 |
| `DEBUFF_curse` | 诅咒 | 拖入献祭台会额外扣理智 | 献祭等价清除 |

### 14.5 净化方式（四种，不直接使用道具）

#### 方式 A：休息

将调查员卡拖入特定场景卡（如书桌、抽屉、休憩点）→ 占用该场景的探索资源 → 逐步恢复理智。

**交互**：拖拽调查员到场景卡
**消耗**：该场景的探索次数 / 时间
**收益**：+20~30 理智
**限制**：每个场景可休息次数有限

#### 方式 B：对话净化

特定人物对话末尾增加净化选项，消耗一次对话机会换取理智恢复。

**交互**：对话窗口中出现"请求帮助"等选项
**消耗**：一次对话机会（该角色进入冷却）
**收益**：+15~25 理智，清除特定负面卡

#### 方式 C：链式消除

某些负面卡只能被另一张特定负面卡消除，形成消除链。

```
恐惧(Fear) → 愤怒(Anger) → 平静(Calm)
   ↓ 消除        ↓ 消除        ↓ 消除
  需要愤怒卡    需要平静卡    净化完成
```

**流程**：
1. 玩家先承受「恐惧」效果
2. 恐惧在场期间会按固定间隔产生「愤怒」卡片
3. 将愤怒拖到恐惧上 → 二者消除，产生「平静」卡
4. 平静可作为资源用于其他净化

**核心设计**：不是用道具点一下，而是"以毒攻毒"，需要玩家先承受负面效果再寻找解药。

#### 方式 D：献祭等价

将负面卡 + 等价物品拖入献祭台，形成"以物换命"。

**交互**：拖拽「负面卡 + 物品」到献祭卡
**消耗**：负面卡 + 任意物品
**收益**：清除该负面卡（但理智不恢复）
**限制**：献祭本身会消耗一个物品资源，形成资源权衡

### 14.6 核心循环（自我加速螺旋）

```
探索 → 获取收益
  ↓
理智下降（探索本身扣减 + 负面卡持续在场）
  ↓
理智越低 → 定时弹卡间隔越短 → 负面卡密度越高
  ↓
必须消耗回合/资源进行净化
  ↓
净化占用探索时间 → 探索收益减少 → 但阻止了崩溃
  ↓
理智降至 0 → 游戏结束
```

**设计要点**：
- 这是一个**自我加速的螺旋**，天然制造紧张感
- 玩家在"继续探索获取关键卡"和"停下来净化"之间做权衡
- 理智越低，弹卡越快，迫使玩家分心处理
- 高操作水平玩家可以维持理智在安全线，低水平玩家会被螺旋压垮
- 不需要剧情脚本，系统级压迫

### 14.7 与现有系统的关系

| 系统 | 关系 |
|------|------|
| 腐化系统 | 独立并存：腐化是单卡倒计时，理智是全局资源；二者可叠加制造双重压力 |
| 探索系统 | 探索扣理智 + 负面卡掉落入口 |
| 对话系统 | 对话提供净化选项 |
| 献祭系统 | 献祭等价清除负面卡 |
| 调查员卡 | 休息净化依赖于调查员与场景的交互 |

### 14.8 待确定问题

- 理智归零后：立即结束 vs 进入"疯狂"状态再给一次机会？
- 连续净化是否衰减（收益递减）？
- 不同调查员初始理智/恢复速度是否不同？
- 负面卡是否有场上限？
- 定时弹出是否考虑桌面已有负面卡数量（太拥挤时暂停）？
